# 话说清楚 (say-it-well) — 项目全貌

> 开源工具：让不懂代码、不懂英语、不懂 AI 的普通人，通过飞书聊天把模糊想法炼成精准编程提示词，用于 Codex / Claude Code / Trae 等 AI 编程工具。

---

## 一、项目定义

```
输入：普通人一句模糊的中文（"帮我做个记账的"）
  ↓
Hermes 引导式对话（判断真伪需求、追问细节）
  ↓
输出：结构化中英双语精准提示词
  ↓
用户自行丢给 Codex / Claude Code / Trae 执行
```

**不是**：写代码的工具、SaaS 平台、商业产品
**是**：开源 AI 辅助需求分析工具，用户自己配 token，自己部署

---

## 二、核心设计原则

### 1. 信任 > 功能
- 开场白清晰自介："我是提示词助手，不是写代码的"
- 能力透明："我能做网页/小程序/脚本，造不了火箭"
- 每一步让用户看到进展

### 2. 对话质量 > 技术能力
- AI 成功率 60-80%，对普通人来说 60% = 废物
- 降级策略：做不了全栈就做静态页，做不了部署就给 HTML
- 替代方案：主动建议"做不了 A，但类似的 B 我可以帮你"

### 3. 数据闭环
- 每次对话沉淀为"普通人如何描述技术需求"的语料
- 统计高频需求类型，优化追问模板
- 积累"模糊描述 → 精准术语"映射表

---

## 三、核心方法论：反思推理

不是机械查表追问，而是每收到用户一句话，先推理、再行动：

1. **理解这句话意味着什么** — 用户说"卖情绪价值"→ 非标品，不能用淘宝逻辑
2. **推导他没想到但必须面对的问题** — 忽略的前提、隐含的假设、没说出口的困难
3. **转化成他能听懂的问题** — 用他的语言，不用术语

示例：
```
机械模式：
  用户："我想做情绪陪伴网站"
  AI：查表 → "靠什么赚钱？"（还没建立信任先谈钱）

反思推理：
  用户："我想做情绪陪伴网站"
  推理：情绪陪伴的核心是信任，不是功能
  推导：第一印象决定用户走还是留
  追问："一个心情不好的人第一次打开你的网站，看到的第一眼是什么？"
```

## 四、对话流程（6 步）

### 第一步：建立信任
- 自我介绍 + 能力透明 + 降低预期

### 第二步：初始捕获
- 复述确认："我理解你是想做……对吗？"
- 可行性判断：可行/部分可行/不可行

### 第三步：多维度追问（一次只问一个问题）
| 维度 | 追问示例 |
|------|---------|
| 目标用途 | "这个东西主要用来做什么？" |
| 目标用户 | "什么人在什么场景下用？" |
| 使用形式 | "网页、小程序、还是手机 App？" |
| 核心功能 | "最重要的功能是什么？只能选 1-2 个" |
| 盈利方式 | "靠什么赚钱？" |
| 内容来源 | "内容从哪来？" |
| 运营维护 | "做完就放那，还是经常更新？" |
| 数据存储 | "要不要记住用户信息？" |
| 外观参考 | "有没有类似的 App 参考？" |
| 完成标准 | "做到什么程度可以先上线？" |
| 共创机制 | "用户出内容，谁审核？怎么展示？" |

### 第四步：补齐翻译
- 模糊词 → 精准词（"好看的"→"现代简约风格"）
- 填补空缺 + 技术选型建议

### 第五步：回头确认
- 整理总结 → 用户确认无误 → 再输出

### 第六步：双语输出
- 中文 prompt + English prompt + 工具建议 + 注意事项

---

## 五、关键技术决策

1. **飞书入口**：用户通过飞书 Bot 对话，不需安装新 App
2. **多引擎**：针对 Codex / Claude Code / 通用三种风格输出不同 prompt
3. **开源非 SaaS**：用户自己配 token，自己部署，不依赖第三方
4. **中文原生**：用中国人的表达习惯，不是翻译英文思维

## 六、技术架构

```
用户（飞书）
    │
    ▼
飞书 Bot（Hermes Gateway）
    │
    ▼
Hermes Agent
    ├── skill: say-it-well（核心）— 引导对话 + 需求炼化
    ├── skill: requirement-classifier — 真伪需求判断
    ├── skill: tech-mapper — 模糊词 → 技术术语映射
    ├── skill: prompt-builder — 多引擎 prompt 模板生成
    └── memory — 用户偏好、历史项目
```

## 八、关键原则

1. **默认用户什么都不会** — 不假设任何先验知识
2. **先建立信任再问问题** — 用户怕 AI，先让他放松
3. **一次只问一个问题** — 多个问题像审问，普通人会跑
4. **失败是常态，包装要好** — AI 做不出来正常，别让用户觉得自己没说清楚
5. **输出就是终点** — 用户拿到 prompt 就算成功
6. **中文原生** — 用中国人的表达习惯，不是翻译英文思维

---

## 九、常见错误（避坑）

- **一次问多个问题** — 最大的坑，普通人看到一堆问号直接放弃
- **过早追问技术细节** — 还没建立信任就问"要不要后端数据库"
- **用术语解释术语** — "CMS"、"SPA"、"SSR" 都是致命词
- **不给选项让用户干想** — 用户说"不知道"时立刻给 2-3 个选项
