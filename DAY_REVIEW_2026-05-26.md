## 二、反复出现的问题

### 🔴 问题 1：技能规则需要多次打补丁才能生效
**现象**：同一个问题（行业追问）改了 4 次才勉强到位——加维度→改智能判断→强化过渡语→提到铁律位置。
**根因**：模型不总是遵守 SKILL.md 里的每一条规则。长文本中，具体指令容易被淹没。
**建议**：关键铁律（行业、复述确认）只用一句话写在 channel_prompt 里，不在 SKILL.md 里长篇展开。

### 🔴 问题 2：飞书平台限制反复踩坑
**现象**：clarify 卡死 → 禁用；/skill 不支持 → 用 auto_skill；模型切换不生效 → 补 .env key + session 锁定。
**根因**：飞书 adapter 是 Hermes 所有平台中功能最弱的，每次功能测试都会发现新的缺失。
**建议**：飞书专属功能清单——明确哪些能做、哪些不能做，避免每次都重新踩坑。

### 🟡 问题 3：config.yaml 和 .env 双配置体系
**现象**：DEEPSEEK_API_KEY 只在 config.yaml 的 custom_providers 里，飞书 session 解析不到。
**根因**：Gateway 和 CLI 读配置的路径不完全一致。
**建议**：API key 两边都放，.env + config.yaml 双写。

### 🟡 问题 4：每次改 skill 都要重启 Gateway
**现象**：今天重启了约 15 次 Gateway。每次改完 SKILL.md → 重启 → 验证，循环。
**根因**：auto_skill 在 session 创建时加载，已创建的 session 读到的是旧版。
**建议**：开发阶段写一个热重载机制或至少接受这个流程。

### 🟡 问题 5：对话质量靠运气
**现象**：同样的问题（客服机器人），不同 session 质量差异大——一个深挖了行业和派单流程，另一个 6 轮就结束了。
**根因**：用户回答的详细程度决定了 bot 追问的深度。用户回答简短→bot 快速收工。
**建议**：在 skill 里加最低追问轮数下限（比如至少 5 轮），防止过早结束。

## 四、经验教训

1. **先查源码再改配置**——平台适配问题不是配置问题是代码问题（飞书 adapter 缺失功能）
2. **关键铁律写两遍**——SKILL.md 里写一遍，channel_prompt 里再写一遍短版
3. **一个维度一个问题**——不要在同一个补丁里修多个问题，分开提测
4. **非技术用户预期管理**——他们不知道拿到 prompt 后怎么做，上路指南是必需品不是锦上添花
5. **飞书是功能最弱的 Hermes 平台**——所有新功能先在其他平台验证，再适配飞书
